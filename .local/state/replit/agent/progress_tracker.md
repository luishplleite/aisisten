## Correção CRÍTICA do Sistema de Edição de Restaurantes - Admin (2025-10-11 - 17:06)
[x] Identificação do erro HTTP 500 via logs do servidor ✅
[x] Análise do schema do banco de dados (bd.sql) ✅

[x] Identificação dos dois campos distintos: status vs subscription_status ✅
[x] Implementação de mapeamento inteligente no frontend (admin.html) ✅
[x] Correção do backend para aceitar subscription_status (server.js) ✅
[x] Correção do bug de sobrescrita automática (architect review 1) ✅
[x] Validação final com architect (PASS) ✅
[x] Atualização da documentação (replit.md + progress_tracker.md) ✅

**🐛 Problema Raiz Identificado:**
- ❌ **Erro:** HTTP 500 ao salvar alterações de restaurantes
- ❌ **Causa:** Violação de CHECK constraint "restaurants_status_check"
- ❌ **Motivo:** Sistema misturava dois campos com regras diferentes:
  * Campo `status`: aceita APENAS 'active', 'inactive', 'suspended'
  * Campo `subscription_status`: aceita 'trial', 'active', 'expired', 'cancelled'
- ❌ **Bug 1:** Frontend enviava 'trial' para campo `status` (violação!)
- ❌ **Bug 2:** Backend sobrescrevia `subscription_status='active'` automaticamente

**✅ Correções Implementadas:**

1. **Frontend (public/admin.html):**
   - Lógica de mapeamento inteligente em `saveRestaurantChanges()`:
     * 'trial' → status='active' + subscription_status='trial'
     * 'active' → status='active' + subscription_status='active'
     * 'suspended' → status='suspended' + subscription_status mantém valor
     * 'cancelled' → status='inactive' + subscription_status='cancelled'
   - Envia AMBOS os campos separadamente para o backend

2. **Backend (server.js - linhas 2098-2171):**
   - Adicionado `subscription_status` aos parâmetros aceitos
   - Adicionado suporte para salvar `subscription_status` no updateData
   - **Correção crítica:** subscription_status só é definido como 'active' SE não for fornecido
   - Condicional: `if (!subscription_status) { updateData.subscription_status = 'active'; }`

3. **Validações:**
   - ✅ Architect Review 1: Identificou bug de sobrescrita
   - ✅ Architect Review 2 (Final): PASS - Constraints respeitadas

**Mapeamento Completo de Campos:**
- `name` → restaurants.name

- `owner_phone` → restaurants.owner_phone (corrigido de 'phone')
